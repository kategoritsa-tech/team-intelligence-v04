# Changelog

История изменений проекта **Team Intelligence**.

Формат версий:

```text
vMAJOR.MINOR.PATCH
```

- `MAJOR` — крупные изменения логики продукта;
- `MINOR` — новая функциональность;
- `PATCH` — небольшие улучшения, фиксы, документация.

---

## v0.5.4

Добавлен анализ пропущенных daily-отчетов.

### Добавлено

- Новый лист Excel `Пропуски отчетов`.
- Поиск рабочих дней без daily-отчета.
- Отображение пропусков по авторам.
- Комментарий к каждому найденному пропуску.
- Новый лист добавлен в управленческий порядок Excel.

### Особенности

- Пропуски считаются только по рабочим дням.
- Пропуски не считаются до первого отчета автора.
- Пропуски не считаются после последнего отчета автора.
- Лист помогает увидеть нерегулярность daily-отчетности.

## v0.5.3

Улучшена оценка качества daily-отчетов.

### Исправлено

- Улучшена нормализация строк с заголовками:
  - `-вчера`
  - `-сегодня`
  - `-проблемы/блокеры`
- Исправлены ложные замечания:
  - `нет блока 'Вчера'`
  - `нет блока 'Сегодня'`
  - `нет блока 'Проблемы/Блокеры'`
- `Проблемы: нет` больше не считается реальной проблемой.
- `Блокеров нет` больше не считается реальной проблемой.
- Замечание `нет Jira-задач` заменено на более нейтральное:
  - `нет привязки к задаче или объекту работ`

### Изменено

- Анализатор мягче оценивает отчеты без Jira, если в них есть конкретная работа.
- Улучшена логика определения формулировок с малой конкретикой.
- Уменьшено количество ложных срабатываний в замечаниях.

## v0.4.1

Добавлен исходный текст daily-отчета в выгрузки.

### Добавлено

- Колонка `Текст отчета` в Excel.
- Колонка `Текст отчета` в CSV `daily_metrics.csv`.

### Зачем

Лид может видеть не только оценку анализатора, но и исходный текст отчета, чтобы:

- вручную проверить выводы;
- разобрать конкретные примеры с сотрудником;
- понять контекст замечаний;
- не опираться только на автоматическую оценку.

## Принципы развития

Проект развивается как инструмент для PM / лида, а не как абстрактная Telegram-аналитика.

Приоритеты развития:

1. Польза для управленческого анализа.
2. Быстрый поиск проблемных отчетов.
3. Возможность ручной проверки выводов.
4. Прозрачность метрик.
5. Безопасность реальных данных сотрудников.
6. Минимум лишней архитектуры до появления реальной необходимости.

## Что не коммитить

Сгенерированные отчеты могут содержать реальные данные сотрудников.

Не коммитить:

```text
src/reports/team_intelligence.xlsx
src/reports/team_intelligence.pdf
src/reports/daily_metrics.csv
src/reports/authors.csv
src/reports/report.txt
```
