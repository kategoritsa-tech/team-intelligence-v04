# Team Intelligence — Architecture Decision Record (ADR)

> В этом документе фиксируются все архитектурные и технические решения проекта.
>
> Принятое решение считается неизменным до появления объективной причины для его пересмотра.
> При изменении старое решение не удаляется, а помечается как устаревшее с указанием причины.

# ADR-001 — Язык разработки

**Дата:** 28.06.2026

## Статус

✅ Принято

## Решение

Основной язык разработки — **Python**.

## Причины

- Богатая экосистема для анализа данных.
- Отличная поддержка работы с HTML.
- Простое создание PDF, Excel и графиков.
- Хорошая интеграция с AI.
- Большое количество библиотек для NLP и машинного обучения.

## Последствия

Все новые модули разрабатываются на Python.

# ADR-002 — Основной источник данных

**Дата:** 28.06.2026

## Статус

✅ Принято

## Решение

На первом этапе используется **HTML-экспорт Telegram**.

## Причины

- Не требует доступа к API.
- Подходит для анализа исторических данных.
- Позволяет быстрее создать MVP.
- Не зависит от изменений Telegram API.

## План развития

В будущих версиях будет добавлен импорт через Telegram API (Telethon) без изменения слоя аналитики.

# ADR-003 — Архитектурный принцип

**Дата:** 28.06.2026

## Статус

✅ Принято

## Решение

Разработка ведется короткими спринтами.

Каждый спринт обязан завершаться:

- работающей функцией;
- коммитом;
- push в GitHub;
- обновлением документации.

## Причины

- Минимизация технического долга.
- Постоянно рабочий продукт.
- Возможность отката к любой стабильной версии.

# ADR-004 — Работа с данными

**Дата:** 28.06.2026

## Статус

✅ Принято

## Решение

Реальные данные пользователей не хранятся в репозитории Git.

## Причины

- Конфиденциальность.
- Безопасность.
- Возможность публикации проекта.

## Последствия

Папка `data/` исключена из Git через `.gitignore`.

# ADR-005 — Принцип автоматизации

**Дата:** 28.06.2026

## Статус

✅ Принято

## Решение

Любое действие, которое можно автоматизировать без существенного усложнения проекта, должно быть автоматизировано.

## Причины

- Снижение количества ручных операций.

- Уменьшение вероятности ошибок.

# ADR-006 — Принцип стабильной архитектуры

**Дата:** 28.06.2026

## Статус

✅ Принято

## Решение

Архитектура проекта не изменяется без объективной необходимости.

Объективными причинами считаются:

- изменение бизнес-требований;
- выявленное архитектурное ограничение;
- существенное улучшение, подтвержденное анализом.

## Причины

- Предотвращение бесконечного рефакторинга.
- Стабильность разработки.
- Предсказуемость проекта.
## 2026-06-28

### Архитектура проекта

Принято разделить приложение на независимые слои:

- parser — чтение и парсинг Telegram HTML
- analytics — вычисление статистики
- reports — отображение результатов
- __main__ — точка входа и orchestration

Причина:
Соблюдение принципа Single Responsibility и упрощение дальнейшего развития проекта.
